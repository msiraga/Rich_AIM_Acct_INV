# AI Integration

## Architecture

The AI module integrates with **Ollama** — a local LLM runtime — to provide:

```
AIService
├── config: AIConfig
├── ollama: Option<ollama_rs::Ollama>   ← Lazily connected
└── orchestrator: Arc<Mutex<AgentOrchestrator>>

TextClassifier                          TextExtractor
├── ai_service: Arc<Mutex<AIService>>    ├── ai_service: Arc<Mutex<AIService>>
├── classify(text, categories)           └── extract(text, schema)
└── → String (category)                      → serde_json::Value
```

## Configuration

```rust
pub struct AIConfig {
    pub enabled: bool,                      // Default: true
    pub ollama_url: String,                  // Default: http://localhost:11434
    pub default_model: String,               // Default: "llama3"
    pub embedding_model: String,             // Default: "llama3"
    pub classification_model: String,        // Default: "llama3"
    pub extraction_model: String,            // Default: "llama3"
    pub timeout: u64,                        // Default: 30s
    pub max_tokens: u32,                     // Default: 2048
    pub temperature: f32,                    // Default: 0.7
}
```

Environment variable overrides: `AI_ENABLED`, `OLLAMA_URL`, `AI_DEFAULT_MODEL`, etc.

## Capabilities (Designed)

### 1. Document Classification

```rust
classify_document(document) → String
```

Prompts the LLM to categorize a document into: Invoice, Receipt, BankStatement, Check, PurchaseOrder, TaxForm, Contract, Other.

### 2. Data Extraction

```rust
extract_data(document, extraction_type) → serde_json::Value
```

Three specialized extraction schemas:
- **Invoice extraction:** vendor, invoice_number, date, due_date, total_amount, tax_amount, line items
- **Receipt extraction:** merchant, transaction_date, total_amount, tax_amount, payment_method, line items
- **Bank statement extraction:** bank_name, account_number, statement_date, starting/ending balance, transactions array

### 3. Text Embeddings

```rust
generate_embeddings(text) → Vec<f32>
```

For semantic search across documents and transactions.

### 4. Transaction Analysis

```rust
analyze_transaction(description) → serde_json::Value
```

Returns: `{ is_suspicious, confidence, issues[], recommendations[] }`

### 5. Account Categorization

```rust
suggest_account_category(name, description) → String
```

Suggests Asset/Liability/Equity/Revenue/Expense for a new account.

## Current Limitations

1. **No actual OCR pipeline** — `extract_text()` returns a placeholder. Real document processing would need Tesseract, pdf-extract, or a vision-capable LLM.
2. **Binary document content is never interpreted** — the LLM prompts say "(binary data, N bytes)" rather than extracting text from PDFs/images first.
3. **No embedding storage or vector search** — embeddings are generated but never indexed.
4. **No model management** — assumes models are already pulled in Ollama.
5. **No prompt engineering iteration** — prompts are naive single-shot templates.
6. **No confidence thresholds or fallback logic** — if the LLM returns malformed JSON, it's silently returned as a string.

## Dependency

The crate depends on `ollama-rs = "0.1"` — a community Rust client for Ollama's HTTP API. Ollama itself must be installed and running separately.
